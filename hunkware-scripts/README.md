# HunkWare Script Reader

An internal tool for the Merrillville Client Communication team. Upload a
College HUNKS work order PDF, review the auto-extracted job details, and
generate one of four call scripts (Welcome, Confirmation, Midpoint
Confirmation, Happy/Post-Job) in clean, ready-to-read English — styled to
match the HunkWare CRM.

## What this is — and isn't

- **Is:** a standalone tool you run yourself. You upload a work order PDF
  (exported from HunkWare) and it extracts the job fields and generates
  scripts from them.
- **Isn't:** a live integration with HunkWare. It does not log into
  HunkWare, scrape pages, or pull data automatically. There's no API
  connection — if HunkWare ever provides API access, the parser layer
  (`backend/parser.py`) can be swapped for a live data source without
  touching the script-generation or UI layers.

## How it works

1. **Upload** a work order PDF (the kind HunkWare exports per job).
2. The server reads the embedded text layer (not OCR) and extracts:
   Client Name, Service Date, Service Type, Arrival Window, Origin
   Address, Destination Address, # of HUNKS, Estimated Hours, $ Amount,
   and the inventory/item list.
3. **Review** the extracted fields — anything the parser couldn't find is
   flagged in red so you can fill it in manually.
4. **Pick a script type** and generate. The script appears in a
   readable, copy-pasteable block.

## Script wording source

All four script templates are transcribed directly from:
- `Welcome_and_Confirmation_Call_Scripts.pdf` (Welcome + Confirmation,
  separate versions for Move/Move Labor vs. Junk Removal)
- `post_job_call_guide_revised.pdf` (Happy/Post-Job call)

**Midpoint Confirmation** has no standalone script in the source
material — the source PDF only defines a *timing rule* ("add a midpoint
check-in 2 weeks before service" for jobs booked more than 2 weeks out).
The current Midpoint template reuses the Confirmation Call structure
with lighter check-in framing, clearly labeled in the output as a
placeholder. Edit `generate_midpoint_call()` in
`backend/scripts_engine.py` once final wording is available.

To change any script wording, edit the corresponding function in
`backend/scripts_engine.py` — it's plain Python f-strings, no template
engine to fight with.

## Local development

```bash
cd backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

export SECRET_KEY=$(python3 -c "import secrets; print(secrets.token_hex(32))")
export ADMIN_EMAIL=you@yourcompany.com
export ADMIN_PASSWORD="choose-a-strong-password"

python3 app.py
```

Visit `http://127.0.0.1:5000`, log in with the admin credentials above.
Once logged in, generate invite codes for teammates from the user menu
(`/admin/invite`) — there's no open public signup.

## Production deployment (Docker — recommended)

1. Copy `.env.example` to `.env` and fill in real values:
   ```bash
   cp .env.example .env
   ```
   Generate a real `SECRET_KEY`:
   ```bash
   python3 -c "import secrets; print(secrets.token_hex(32))"
   ```

2. Build and run:
   ```bash
   docker compose up -d --build
   ```

3. The app is now running on port 5000 of the host. Put it behind a
   reverse proxy (nginx, Caddy, or your cloud provider's load balancer)
   with **HTTPS terminated there** — see the security notes below.

4. Log in with `ADMIN_EMAIL` / `ADMIN_PASSWORD` from your `.env`, then
   generate invite codes for the rest of the team.

Job data (the SQLite database with user accounts) persists in a Docker
volume (`script-reader-data`) so it survives container restarts/rebuilds.

## Production deployment (without Docker)

```bash
cd backend
pip install -r requirements.txt
export SECRET_KEY=...
export ADMIN_EMAIL=...
export ADMIN_PASSWORD=...
export FLASK_ENV=production
gunicorn --bind 0.0.0.0:5000 --workers 2 --timeout 60 "app:create_app()"
```

Put this behind a reverse proxy with HTTPS (nginx/Caddy). Do not expose
gunicorn directly to the internet without TLS in front of it.

## Security notes

- **Passwords** are hashed with werkzeug's PBKDF2/scrypt — never stored
  in plaintext.
- **Sessions** use HttpOnly, SameSite=Lax cookies; `Secure` is enabled
  automatically when `FLASK_ENV=production`. This means cookies will
  only be sent over HTTPS in production — make sure your reverse proxy
  actually terminates TLS, or login will silently fail.
- **Login throttling**: 5 failed attempts per IP+email locks out further
  attempts for 5 minutes. This is in-memory and per-process; for a
  larger deployment swap it for Flask-Limiter backed by Redis.
- **No public signup**: new accounts require a one-time invite code
  generated by an existing admin (`/admin/invite`).
- **Uploads**: restricted to `.pdf`, capped at 10MB, processed entirely
  in memory and never written to disk. No PDF content is persisted
  server-side after the request completes — only what you choose to
  generate into a script (which itself isn't stored either; nothing in
  this version saves job history to the database).
- **Security headers** are set on every response: `X-Content-Type-Options`,
  `X-Frame-Options: DENY`, `Referrer-Policy`, and a baseline
  `Content-Security-Policy`.
- **Secrets**: never commit `.env`. `.gitignore` is already configured
  to exclude it, the SQLite database, and Python caches.

## What's deliberately not included (yet)

- **Job history / audit log** — nothing persists after you close the
  tab. If your team wants a record of which scripts were generated for
  which jobs, that's a small addition to `views.py` + a new DB model —
  flag it and it can be added.
- **Live HunkWare integration** — see "What this is — and isn't" above.
- **Bulk PDF upload** — currently one work order at a time.
- **Role-based permissions beyond admin/non-admin** — every logged-in
  user can generate any script; only admins can issue invite codes.
